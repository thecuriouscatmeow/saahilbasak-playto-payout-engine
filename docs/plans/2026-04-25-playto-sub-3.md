## Goal
Implement the payout creation critical path end-to-end: `SELECT FOR UPDATE` lock, balance check, payout + hold insert, idempotency three-case handling, and the state-machine transition primitive. Six integration tests must pass.

## Acceptance Criteria
1. `POST /api/v1/payouts` with sufficient balance → 201 + payout object, 1 `Payout` row, 1 `hold` `Transaction` row.
2. `test_concurrency_tier1` — ₹100 merchant, 2 simultaneous ₹60 POSTs → exactly 1 success (201), 1 reject (422), `available_paise` never negative.
3. `test_concurrency_tier2` — ₹300 merchant, 10 simultaneous ₹60 POSTs → exactly 5 succeed, 5 reject, sum of `hold` rows = ₹300.
4. `test_idempotency_replay` — same key + same body twice → 1 `Payout` row, both responses byte-identical.
5. `test_idempotency_in_flight` — second arrives during first → 202 or stored response, never 2nd `Payout`.
6. `test_idempotency_payload_mismatch` — same key, different body → 409 `key_reused_with_different_body`.
7. `test_state_machine_guards` — every illegal transition raises `InvalidStateTransition`.

### M3.3 — `CreatePayoutService` happy path
- `apps/payouts/services/create_payout.py::CreatePayoutService`:
  - Constructor freezes inputs (`merchant_id, amount_paise, bank_account_id, idempotency_key, request_hash, raw_body`).
  - `execute() -> (status_code, response_body)`:
    - Insert idempotency record with `state=in_progress`. If created, proceed to atomic block.
    - Atomic block: `merchant_repo.lock_for_update`, `get_balance_breakdown`, balance check, `payout_repo.create_with_hold`, serialize, `idempotency_repo.update_with_response(state=completed, status=201)`.
    - Outside atomic: enqueue `tasks.process_payout.delay(payout.id)` (stub task — sub-4 implements; for now a no-op `@shared_task`).
- **Test:** `tests/integration/test_create_payout_happy_path.py` — POSTs valid request, asserts 201, asserts 1 Payout + 1 hold txn, asserts task enqueued (mocked / inspecting `CELERY_TASK_ALWAYS_EAGER` test setting).
- **Commit:** `feat(service): CreatePayoutService happy path`

### M3.4 — Three-case idempotency + 422 error path
- Extend `CreatePayoutService.execute()` per spec §6 STEP 2 + ERROR PATHS:
  - `_handle_existing_record(record, incoming_hash, body)`:
    - hash differs → `IdempotencyPayloadMismatch` (Case C).
    - state=`completed` → return stored `(status_code, response_body)`.
    - state=`in_progress` → poll loop 5×200ms; if becomes completed return stored, else 202 `{status, payout_id, retry_after_ms}`.
  - On `InsufficientBalance`: catch outside atomic; in a fresh transaction, `idempotency_repo.update_with_response(state=COMPLETED, status=422, body={...})`. Then re-raise so the API layer maps to 422.
- **Test:** `tests/integration/test_idempotency_replay.py`, `test_idempotency_in_flight.py`, `test_idempotency_payload_mismatch.py`. The in-flight test uses `threading` to start request A (mocked to sleep inside the atomic block), kicks off request B during the sleep, asserts B's behavior.
- **Commit:** `feat(idempotency): three-case handling + 422 error-path record snapshot`

### M3.5 — DRF API + exception handler
- `apps/payouts/api/serializers.py`: `CreatePayoutRequest` (DRF Serializer with `amount_paise`, `bank_account_id`), `PayoutResponse`.
- `apps/payouts/api/exceptions.py`: custom DRF exception handler mapping:

  - `InvalidStateTransition → 409` with `{error, payout_id, from, to}`.

- `apps/payouts/api/views.py`:

  - `PayoutListView`, `PayoutDetailView`, `PayoutEventsView`.
- `config/settings/base.py` set `REST_FRAMEWORK["EXCEPTION_HANDLER"]` to the custom one.
- **Test:** `tests/integration/test_payout_api_contracts.py` exercising every status code from spec §12.1.
- **Commit:** `feat(api): payout endpoints + DRF exception handler for domain errors`

### M3.6 — Concurrency tests
- `tests/integration/test_concurrency_tier1.py`:
  - Seed merchant with ₹100 credit, two bank accounts (or one).
  - `ThreadPoolExecutor(max_workers=2)`, `threading.Barrier(2)`. Each thread: open its own DB connection (close-old-connections trick or `connection.close()` per thread to force fresh), wait on barrier, POST `{amount_paise: 6000, bank_account_id: ...}` with distinct `Idempotency-Key`.
  - Assert: results are exactly `{201, 422}` set; `Payout.objects.count() == 1`; `merchant_repo.get_balance_breakdown(...).available_paise == 4000` (or 10000 if rejected, depending on which won — but balance must equal `100 - winner_amount` and never be negative).
- `tests/integration/test_concurrency_tier2.py`:
  - ₹300 credit, 10 threads × ₹60 each. Assert exactly 5 successes, 5 422s, sum of `hold` rows = ₹300, available = 0.
- These tests run against real PG; `pytest-django` with `transaction=True` and `django_db_serialized_rollback`.
- **Commit:** `test(concurrency): tier-1 + tier-2 concurrent payout tests`
