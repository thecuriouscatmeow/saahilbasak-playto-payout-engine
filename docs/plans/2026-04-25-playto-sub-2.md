## Goal
Implement the read-side ledger: the canonical balance aggregation query, the typed transaction insert wrappers, and the merchant-facing read APIs. No payouts yet.

## Acceptance Criteria
1. `GET /api/v1/merchants` returns seeded merchants.
2. `GET /api/v1/merchants/{id}/balance` returns `{available_paise, held_paise, total_credits_paise}` matching ledger arithmetic.
3. `GET /api/v1/merchants/{id}/transactions` returns paginated ledger items with all 4 types representable.
4. `GET /api/v1/merchants/{id}/bank_accounts` returns seeded bank accounts (account number masked: `XXXX1234`).
5. `test_balance_aggregation` passes — fabricates rows of all 4 types, asserts `available`/`held`/`total_credits` formulas hold per spec §5.1.
6. Balance computation is verifiably DB-side: no `for txn in queryset` summation in repository code.

## Milestones

### M2.1 — `merchant_repo` (TDD)
- **RED:** `tests/integration/test_balance_aggregation.py` — for a fabricated merchant with credits / holds / releases / debits in mixed orders, assert `get_balance_breakdown()` returns the spec §5.1 values exactly. Run → fails (function doesn't exist).
- **GREEN:** implement `apps/payouts/repositories/merchant_repo.py`:
  - `lock_for_update(merchant_id) -> Merchant`: returns merchant via `Merchant.objects.select_for_update().get(id=merchant_id)`. Must be called inside `transaction.atomic()`.
  - `get_balance_breakdown(merchant_id) -> BalanceBreakdown` (small dataclass with `available_paise`, `held_paise`, `total_credits_paise`): single Django ORM aggregation matching the SQL in spec §5.1, using `Sum(Case(When(type='credit', then=F('amount_paise')), default=Value(0)))` etc.
  - **No** Python iteration over fetched rows.
- Re-run test → passes.
- **Commit:** `feat(repo): merchant_repo with single-aggregation balance query`

### M2.2 — `transaction_repo` typed insert wrappers
- `apps/payouts/repositories/transaction_repo.py`:
  - `insert_credit(merchant_id, amount_paise, reference=None) -> Transaction`
  - `insert_hold(payout, amount_paise, reference=None) -> Transaction`
  - `insert_release(payout, amount_paise, reference=None) -> Transaction`
  - `insert_debit(payout, amount_paise, reference=None) -> Transaction`
- Each wrapper sets the correct `type` and (for non-credit) `payout_id`. `merchant_id` derived from `payout.merchant_id` for non-credits.
- **Test:** add cases to `test_balance_aggregation.py` exercising each wrapper; assert resulting balance matches expected.
- **Commit:** `feat(repo): transaction_repo typed insert wrappers (credit/hold/release/debit)`

### M2.3 — `DashboardService`
- `apps/payouts/services/dashboard.py::DashboardService.get_for_merchant(merchant_id)` returning a dict bundling balance breakdown + N most recent transactions. Used by the dashboard endpoint if we expose it; for now the API uses balance + transactions endpoints separately.
- **Test:** unit-style test asserting structure.
- **Commit:** `feat(service): dashboard service bundling balance + recent ledger`

### M2.4 — Merchant + balance + transactions APIs
- `apps/merchants/api/{views,serializers,urls}.py`:
