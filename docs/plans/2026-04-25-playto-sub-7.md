## Acceptance Criteria
1. `docker compose up -d --build` boots seeded app + worker + beat + frontend; dashboard works against the bundled backend.
2. Railway deployment URL serves the same app, with PG and Redis attached.
3. `Makefile` targets all functional: `up`, `down`, `seed`, `test`, `stress`, `reconcile`, `logs`.
4. `README.md`: 5-line quickstart + architecture diagram + test instructions + deployed URL.
5. `EXPLAINER.md`: 9 sections per spec §17, every code snippet pasted from real repo paths.
6. `EXPLAIN ANALYZE` output included for the balance aggregation query and the locking SELECT.

## Milestones

### M7.1 — Backend Dockerfile
- `backend/Dockerfile`:
  - Base: `python:3.12-slim`.
  - Install build deps + `psycopg[binary]`.
  - Multi-stage: builder installs deps to a venv, runtime copies venv + code.
  - Entrypoint: `gunicorn config.wsgi:application --bind 0.0.0.0:8000 --workers 3` (overridden by compose for worker/beat).
- **Test:** `docker build -t playto-backend backend/` succeeds.
- **Commit:** `feat(deploy): backend Dockerfile`

### M7.2 — Frontend Dockerfile + nginx
- `frontend/Dockerfile`:
  - Stage 1: `node:20-alpine`, `npm ci`, `npm run build`.
  - Stage 2: `nginx:alpine`, copy `dist/` → `/usr/share/nginx/html`, custom `nginx.conf` with SPA fallback (`try_files $uri /index.html`) and `/api/` proxy_pass to backend service.
- `frontend/nginx.conf` checked in alongside Dockerfile.
- **Test:** `docker build -t playto-frontend frontend/` succeeds.
- **Commit:** `feat(deploy): frontend Dockerfile + nginx SPA + api proxy`

### M7.3 — `docker-compose.yml`
- Services:
  - `db`: `postgres:16-alpine`, named volume, healthcheck (`pg_isready`).
  - `redis`: `redis:7-alpine`, healthcheck.
  - `web`: build from `backend/`, depends_on db (healthy) + redis. Env: `DATABASE_URL`, `REDIS_URL`, `DJANGO_SECRET_KEY`, `DJANGO_DEBUG=0`, `ALLOWED_HOSTS=*`. Command: `sh -c "python manage.py migrate && python manage.py seed && gunicorn ..."`.
  - `worker`: same image as web, command `celery -A config worker -l info --concurrency=2`.
  - `beat`: same image, command `celery -A config beat -l info`.
  - `frontend`: build from `frontend/`, ports `5173:80`, depends_on web.
- Default network shared across services.
- **Test:** `docker compose up -d --build`; wait for healthchecks; hit `http://localhost:5173` → dashboard works; `docker compose down -v`.
- **Commit:** `feat(deploy): docker-compose with postgres + redis + web + worker + beat + frontend`

### M7.7 — `EXPLAINER.md` (the graded artifact)
Nine sections per spec §17. Each section: 1–3 paragraphs of prose + 1 code block pasted from real repo paths. **Required references:**

1. **Architecture at a Glance** — ASCII diagram. Explain layer responsibilities; explain why services don't import models directly.
2. **The Ledger** — paste `merchant_repo.get_balance_breakdown()` in full + `EXPLAIN ANALYZE` output (capture from a `psql` session). Explain the 4-type model and why no joins.
3. **The Lock** — paste `CreatePayoutService.execute()`'s atomic block. Explain `SELECT FOR UPDATE` row lock; reference `tests/integration/test_concurrency_tier1.py` and `tier2`.
4. **The Idempotency** — paste `idempotency_repo.insert_or_get_by_key()` and the `_handle_existing_record()` branch in `CreatePayoutService`. Walk Cases A/B/C. Explain the `request_hash` motivation.
5. **The State Machine** — paste `payout_repo.transition()`. Explain the WHERE-clause guard; show `LEGAL` set; explain why `failed→completed` is structurally unreachable.
6. **Atomic Refund** — paste the `transition(frm=PROCESSING, to=FAILED, on_apply=lambda: txn_repo.insert_release(...))` call site. Explain "state and money commit together."
7. **Failure Recovery** — paste `payout_repo.claim_stale_with_skip_locked()` and the sweeper service's loop. Cover crash-after-mark, duplicate-delivery, bank-hang, max-attempts.
8. **Observability** — paste a real captured log trace (one payout end-to-end, output of `jq 'select(.payout_id == "...")' logs.jsonl`). Show the `correlation_id` threading.
9. **AI Audit** — one real example. Paste the wrong AI suggestion (e.g., `m = Merchant.objects.get(id=mid); if m.balance >= amt: …` outside a lock; or `release` insert in a separate `transaction.atomic()` from the `transition` call). Paste what replaced it. Explain why.
